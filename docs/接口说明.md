# 接口说明

## 1. 启动与访问

```powershell
.\venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
python main.py
```

Swagger 地址：

```text
http://127.0.0.1:8000/docs
```

如 `.env` 中配置了 `APP_API_KEY`，调用接口时需要传入请求头：

```text
X-API-Key: <APP_API_KEY>
```

LLM Provider 统一使用 OpenAI-compatible 协议。默认读取 `OPENAI_API_KEY`、`OPENAI_BASE_URL` 和 `MODEL_*`；如果本地只配置了 `DEEPSEEK_API_KEY`、`DEEPSEEK_BASE_URL`，客户端也会按 OpenAI-compatible 方式读取，不需要 Anthropic/Claude Provider。

## 2. 第一阶段验收接口

第一阶段只验收数据源读取、手工数据导入、RAG 知识库重建、证据检索和基础看板数据读取。多 Agent、Skill、报告、定时任务和部署接口属于后续阶段。

### 2.1 健康检查

```http
GET /
```

预期返回：

```json
{
  "status": "ok"
}
```

### 2.2 查看数据源

```http
GET /sources
```

验收要点：

- 返回列表不为空。
- 每个数据源包含 `source_id`、`source_type`、`dimension`、`enabled`。
- 手工 CSV 数据源的 `source_type` 应为 `csv`。

### 2.3 导入手工 CSV

```http
POST /ingest/csv
Content-Type: application/json
```

请求示例：

```json
{
  "path": "data/raw/hema_price_manual.csv"
}
```

验收要点：

- `count` 大于 0。
- `data/processed/intelligence_records.csv` 被写入。
- 接口会同步触发 RAG 索引重建。

### 2.4 重建 RAG 知识库

```http
POST /rag/rebuild
```

预期返回包含：

```json
{
  "vector_store": "Chroma",
  "collection": "competitor_intelligence_baai_bge_m3",
  "embedding": {
    "provider": "huggingface",
    "model_name": "BAAI/bge-m3"
  }
}
```

验收要点：

- `chunks` 大于 0。
- `data/processed/rag_index.json` 存在。
- `chroma_db/` 存在并可被后续检索使用。
- RAG 默认使用 Hugging Face `BAAI/bge-m3`，collection 名会带上模型名，避免后续更换商品或 embedding 模型时和旧向量维度冲突。

### 2.5 检索证据片段

```http
GET /rag/search?query=山东寿光黄瓜 河北黄瓜 批发价 到货价 价差&competitor=山东寿光黄瓜&dimension=price&top_k=3
```

单条结果应包含：

```json
{
  "chunk_id": "sgc001-0",
  "record_id": "sgc001",
  "title": "寿光黄瓜批发价小幅下探",
  "text": "山东寿光黄瓜精品货批发价较昨日下调0.2元/kg，到货量增加",
  "source_url": "https://manual.local/wholesale/shouguang-cucumber/sgc001",
  "competitor": "山东寿光黄瓜",
  "dimension": "price",
  "collected_at": "2026-07-01"
}
```

验收要点：

- 返回数组不为空。
- 每条证据必须包含 `source_url`。
- 查询价格数据时 `dimension=price` 能返回价格相关证据。
- 查询舆情数据时 `dimension=sentiment` 能返回投诉、差评、配送慢、不新鲜等证据。

### 2.6 基础看板数据

```http
GET /dashboard/summary
GET /dashboard/comparison
GET /dashboard/risk-tags
```

验收要点：

- `summary` 返回记录列表。
- `comparison` 返回竞品和 `price`、`new_product`、`sentiment` 三类维度统计。
- `risk-tags` 能返回由关键词识别出的风险标签。

推荐第一阶段验收顺序：

1. `GET /`：确认服务启动。
2. `GET /sources`：确认数据源配置可读。
3. `POST /ingest/csv`：导入 `data/raw/hema_price_manual.csv`。
4. `POST /rag/rebuild`：重建知识库。
5. `GET /rag/search`：确认可检索带来源的证据。
6. `GET /dashboard/comparison`：确认维度统计正常。

## 3. 第二天验收接口

第二天对应课件中的 Chains、Agents、Tools、Memory。当前项目不是直接照搬 Anthropic 示例，而是在现有 Skill/Agent 架构中用 OpenAI-compatible Provider 实现同等能力。

### 3.1 查看模型与 RAG 配置

```http
GET /config
```

返回中应包含：

```json
{
  "configured": true,
  "base_url": "https://api.deepseek.com/v1",
  "models": {
    "fast": "deepseek-v4-flash",
    "analysis": "deepseek-v4-pro",
    "report": "deepseek-v4-pro"
  },
  "rag_embedding": {
    "provider": "huggingface",
    "model_name": "BAAI/bge-m3",
    "device": "cpu"
  }
}
```

### 3.2 Tools：采集、导入与证据检索

```http
POST /ingest/manual
POST /ingest/csv
POST /ingest/webpage
POST /ingest/rss
GET /rag/search
```

当前工具层对应 `modules/tools.py`，主要包含手工记录导入、CSV 导入和 RAG 证据检索。网页/RSS 只面向公开可访问内容，不承诺绕过登录、验证码和反爬。

### 3.3 Agents：价格、新品、舆情分析

```http
GET /agents
POST /analyze
```

`GET /agents` 应返回：

```json
[
  {"name": "price_monitor", "dimension": "price", "model_role": "analysis"},
  {"name": "new_product", "dimension": "new_product", "model_role": "analysis"},
  {"name": "sentiment", "dimension": "sentiment", "model_role": "analysis"}
]
```

单 Agent 请求示例：

```json
{
  "competitor": "山东寿光黄瓜",
  "question": "分析山东寿光黄瓜近期批发价和新批次供应信息",
  "agent": "price_monitor",
  "top_k": 6,
  "session_id": "demo-session"
}
```

`agent=all` 时会依次运行价格、新品、舆情 Agent，并生成综合报告。

### 3.4 Chains：RAG 证据到结构化 JSON

第二天链式调用在接口层体现为：

1. `/rag/search` 从 Chroma 检索证据。
2. `/analyze` 调用对应 Agent Prompt。
3. `OpenAICompatibleLLM.chat_json()` 要求模型只返回结构化 JSON。
4. `/report/generate` 或 `/analyze` 的 `agent=all` 整合生成报告。

这一链路对应课件的“检索 -> Prompt -> LLM -> JSON Parser”，实现文件为 `modules/rag_chain.py`、`modules/prompts.py`、`modules/agent_core.py`、`modules/llm_client.py`。

### 3.5 Memory：缓存、Trace 与对话历史

```http
GET /traces
DELETE /memory/cache
GET /memory/conversations/{session_id}
POST /memory/conversations/{session_id}
DELETE /memory/conversations/{session_id}
```

新增对话消息示例：

```json
{
  "role": "user",
  "content": "继续总结刚才山东寿光黄瓜的区域价差和质量风险"
}
```

`/analyze` 传入 `session_id` 后，会自动记录用户问题和助手摘要，便于演示第二天课件里的会话记忆管理。

## 4. 后续功能接口

- `GET /sources`：查看生鲜数据源。
- `POST /sources`：新增或更新数据源。
- `DELETE /sources/{source_id}`：删除数据源。
- `POST /collect/run`：手动运行数据采集任务。
- `GET /collect/scheduler`：查看采集调度状态。
- `GET /collect/logs`：查看采集日志。
- `POST /ingest/manual`：导入单条手工记录。
- `POST /ingest/csv`：导入手工 CSV。
- `POST /ingest/webpage`：导入公开网页。
- `POST /ingest/rss`：导入 RSS。
- `POST /rag/rebuild`：重建 Chroma 知识库。
- `GET /rag/search`：检索生鲜批发采购证据片段，默认 query 为“山东寿光黄瓜 河北黄瓜 批发价 到货价 价差 质量风险”。
- `GET /dashboard/summary`：查看基础动态摘要。
- `GET /dashboard/comparison`：查看竞品维度对比数据。
- `GET /dashboard/risk-tags`：查看风险标签。
- `GET /skills`：查看 Skill 列表。
- `POST /skills/{skill_name}/run`：运行单个 Skill。
- `POST /analyze/multi-agent`：运行价格、供应动态、质量风险、趋势和报告全流程。
- `POST /report/generate`：生成生鲜批发采购区域供应源竞品对标报告。
- `GET /reports`：查看报告列表。
- `GET /reports/{report_name}/preview`：预览报告 Markdown 和 JSON。
- `GET /logs/skill-trace`：查看 Skill 调用日志。
- `GET /traces`：查看 Agent 调用轨迹。
- `DELETE /memory/cache`：清空分析缓存。
- `GET /memory/conversations/{session_id}`：读取指定会话记忆。
- `POST /memory/conversations/{session_id}`：追加会话消息。
- `DELETE /memory/conversations/{session_id}`：清空指定会话记忆。
- `GET /report-schedules`：查看报告计划。
- `POST /report-schedules`：新增或更新报告计划。
- `POST /report-schedules/{schedule_id}/run`：手动运行报告计划。

```json
  "dimensions": ["price", "product", "sentiment", "trend"],
  "report_type": "weekly",
  "provider": "mock",
  "date_range": {
    "start": "2026-07-01",
    "end": "2026-07-06"
  }
}
```
